# Testing Heuristics Reference

## Nielsen's Usability Heuristics (Adapted)

Use these as evaluation lenses for any solution type:

1. **Visibility of system status** — Does the user always know what's happening?
2. **Match between system and real world** — Does it use language and concepts the user already knows?
3. **User control and freedom** — Can the user undo, go back, and exit easily?
4. **Consistency** — Does it behave the same way throughout?
5. **Error prevention** — Does it prevent mistakes before they happen?
6. **Recognition over recall** — Can users see their options rather than remembering them?
7. **Flexibility** — Does it work for both novice and expert users?
8. **Aesthetic and minimalist design** — Is there unnecessary information or complexity?
9. **Error recovery** — When something goes wrong, can the user recover easily?
10. **Help and documentation** — Is help available when needed?

## Design Principle Evaluation Framework

For each design principle from Phase 2:

| Rating | Meaning | Criteria |
|--------|---------|----------|
| **PASS** | Principle is well-served | Solution directly addresses this principle; evidence from prototype |
| **PARTIAL** | Principle is partially addressed | Some aspects served, others missed; specific gaps identified |
| **FAIL** | Principle is not served | Solution contradicts or ignores this principle; specific evidence |

## Feedback Collection Framework

### For Product/UI Prototypes
- **First-click test:** Can the user identify how to start the primary task?
- **Navigation test:** Can the user find what they need within 2-3 clicks?
- **Comprehension test:** Does the user understand what each element does?
- **Emotional response:** How does the interface make the user feel?

### For System/Process Prototypes
- **Flow test:** Does the process make sense from the user's perspective?
- **Handoff test:** Are the transitions between steps/stages clear?
- **Exception test:** What happens when things don't go as planned?
- **Efficiency test:** Are there unnecessary steps or bottlenecks?

### For Business/Strategy Prototypes
- **Value test:** Does the user see clear value in the proposition?
- **Believability test:** Does the business model seem sustainable?
- **Adoption test:** Would the user actually change their behavior to use this?
- **Alternative test:** Why would the user choose this over their current approach?

## Red Flags in Testing

Watch for these signals:
- **Polite praise without specifics** — "It looks nice" = they don't have anything real to say
- **Confusion masked as questions** — "So would I use this when...?" = they don't understand the concept
- **Feature requests** — "It would be great if it also..." = the core isn't compelling enough
- **Silence** — long pauses before responding = processing confusion
- **Workaround descriptions** — "I would probably just..." = they don't trust the solution
